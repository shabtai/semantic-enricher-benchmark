Both arms are the same coding agent answering the same questions against the same raw CSVs. The only difference is whether the agent reads .semantic-enricher/ first. Each question has a planted structural trap; a "PASS" means the agent either gave the correct answer and flagged the trap, or correctly refused when the trap made the question unanswerable as posed.

Where the lift comes from. Enrichment helps most when the agent has the least prior knowledge. What is "silent errors"?

Each question has a single planted structural trap — a way the data is shaped, named, or split that will silently produce a plausible-but-wrong answer if the agent doesn't notice it. The trap categories tested:

TrapWhat it looks likeWhy it's silent
null_sentinelA magic value (-666, 0) used for "missing"Math runs without error; mean drops or flips sign
wrong_grainCounting rows of the wrong table (e.g., admissions vs patients)Always returns a positive integer
same_grain_wide_joinTwo tables that are wide splits of the same logical entityNaive many-to-many join fans the row count out
missing_filterMixed treatment/control or quality flags ignoredNumbers come out, just the wrong cohort
unit_mixingSame column has rows in nM, µM, %, …A single median across mixed units is a meaningless number
system_disjointnessCareVue (2001-08) vs MetaVision (2008-12) ITEMIDs for "heart rate"Querying one ITEMID gives a half-cohort mean
resolution_sumSame loop set called at 1kb / 2kb / 5kb in three filesSumming all three triple-counts
control_as_dataA "_random" file is a permutation null, not a real cohortComparing random-vs-real as if two real conditions
reference_vs_resultA bundled annotation file (e.g., mm39 CTCF .bed) treated as study outputThe file is in the dataset directory
simulation_vs_experimentPolySim simulation outputs read as wet-lab measurementsNumeric columns look like any other measurement
partition_assumptionTwo boolean flags assumed mutually exclusive when actually overlappingMath is correct, claim is wrong

Dataset 1 — chromatin_loops (blind 3/8 → /ask 8/8, +5)

Mouse mESC chromatin loops from Zenodo 14600051, published 2026-04-23 — genuinely post-cutoff for the model's training data. Mix of Mustache loop calls (3 resolutions), Micro-C quantification scores, polymer-simulation outputs, and a bundled mm39 CTCF reference annotation. The trap density is high because the directory contains many overlapping representations of "loops" + a permutation null + simulation outputs + reference annotations all sitting next to each other as similarly-named CSVs.

Why blind failed

  • chrom_se_01 — Without the registry, the agent treated the three Mustache files as independent loop sets and asked clarifying questions, never recognizing that 1kb/2kb/5kb are the same calls at different resolutions.
  • chrom_se_02 — Reported the row count of the _random file as if it were real loops. The filename hint alone wasn't enough; the agent had no notion that "random" here means a permutation null.
  • chrom_se_03 — Picked one file from the three overlapping "final loop" candidates with no acknowledgment that filtered_loops and final_loop_data share keys but differ in annotations.
  • chrom_se_06 — Treated mm39_CTCF_cohesin.bed as study output ("yes, the dataset measured 28,583 CTCF sites"). Missed that mm39 is a genome-build identifier, not an experimental run — this is a bundled reference annotation.
  • chrom_se_08 — Read polysim_calibration_curve as a Micro-C measurement. The word "polysim" should signal "polymer simulation" but on a fresh dataset the agent had no anchor.

How /ask succeeded

The enrichment registry tags each table with its research role — raw_signal, permutation_null, simulation_output, reference_annotation, derivative_summary. Per-table purpose: fields explain provenance ("derived from polymer simulation", "permutation control with random coordinates"). Trap detection had already flagged the resolution-duplication pattern across the three Mustache files. Once the agent reads any of this, the traps become legible — every /ask answer in this dataset cites the role tag or the trap finding before producing the number.

Why blind failed

  • lincs_se_02 — No way to know fact_d is already a rollup of fact_e. Agent did the natural join through fact_e and double-counted instances.
  • lincs_se_03 — Treated the two same-grain wide splits as independent entities and reported a Cartesian fan-out as the answer.
  • lincs_se_07 — Computed a single median(num_d1) = 3.0 across all rows. With masked column names, there's no signal that cat_d1 encodes the unit and pooled medians are meaningless.
  • lincs_se_08 — Counted all 98 rows of entity_a as cell lines. The hierarchy via ref_a1 (parent line) and derivative status was invisible.
  • lincs_se_09 — Computed the right fraction (978/12328) but didn't notice flag_b1 and flag_b2 are overlapping (landmark vs BING gene classes), not mutually exclusive.

How /ask succeeded

The enrichment recovers the semantic identity of each masked column: num_a1 is "donor age in years (-666 = missing)", id_c is "perturbagen FK", cat_d1 is "dose unit code", flag_dm1 is "is_gold quality flag", and so on. The reviewer agent tags flag_b1 with a "competing classifier" warning. Trap detection produces explicit findings — including the new control_subset_filter_required heuristic that flagged the masked PTYPE_* codes in cat_c1 as a treatment/control mix. /ask reads these and operationalizes them in code; only the partition-overlap finding (se_09) was not surfaced strongly enough to trigger a flag in the answer.

Why blind almost wins here

MIMIC-III is one of the most-discussed public datasets in the literature; the agent has clearly seen its schema during training. Blind already knows about the CareVue/MetaVision split, the discharge-location-vs-deathtime gap, and the ICD9_CODE conventions. Of the two blind failures:

  • mimic_se_03 — Refused for an unrelated reason (couldn't find a "glucose" column directly), never identified the actual VALUENUM = 0 placeholder pattern. /ask also fails this — the enrichment surfaces specimen mixing and a urine-dipstick scale issue, but doesn't flag the zero-as-missing convention.
  • mimic_se_07 — Refused due to a coding error rather than recognizing that MICROBIOLOGYEVENTS has one row per (test, organism) pair. /ask reads the grain note and reports both definitions (specimens with growth = 896; organism isolates = 1086) explicitly.

The MIMIC result is the "control" that shows the lift comes from missing prior knowledge, not from generic reasoning improvement. Where the agent already knows the data, enrichment helps only at the margin.

Why blind failed (the mechanism)

Blind failures cluster into two patterns:

  1. Plausible-but-wrong path of least resistance. When the agent doesn't know that fact_d is already a rollup, the natural choice is to join fact_d ⨝ fact_e "for completeness" — and double-count. When it doesn't know that the three Mustache files are the same loops at different resolutions, the natural choice is to sum.
  2. Filename hint isn't enough. Names like mESC_random_loops, mm39_CTCF_cohesin.bed, and polysim_calibration_curve contain the answer in their tokens — but the agent treats them as data without questioning whether the file represents study output, a control, a reference, or a simulation. Filenames carry intent that requires a registry to disambiguate.

In neither case does the agent produce nonsense. The blind answers are articulate — they include caveats, sanity checks, and reasonable-looking numbers. The trap is silent precisely because there is no error message and no internal contradiction; the answer simply doesn't match the meaning of the question.

How /ask succeeded (the mechanism)

The enrichment layer is a precomputed metadata bundle in <data_dir>/.semantic-enricher/. Before answering, /ask reads:

  • CLAUDE.md — natural-language overview of the dataset and per-table summaries
  • semantic_layer/registry.yaml — table index with grain, role, and domain tags
  • semantic_layer/tables/<TABLE>.yaml — per-column meaning, semantic type, null conventions, and research role
  • trap_findings.json — heuristic-detected structural traps (unit-mixing, control-mixing, rollup contamination, …)
  • review_findings.json — reviewer agent's cross-table consistency flags

The agent's reasoning trace shows three patterns of use:

  1. Identity recovery on masked schemas. On LINCS, every /ask answer cites the resolved meaning of the masked columns ("num_a1 = donor age in years per enrichment") before computing.
  2. Trap operationalization. On masked LINCS, the agent reads the control_subset_filter_required finding and explicitly partitions PTYPE_* codes by cardinality (treatment modalities have 61–20K perturbagens; controls have 1–8) before counting.
  3. Role disambiguation. On chromatin, the agent uses the registry's research_role tag to distinguish reference_annotation (mm39 BED) from raw_signal, and simulation_output from experimental measurement, before answering.

The two remaining /ask failures

  • lincs_se_09 (partition_assumption) — Agent computes flag_b1=1 fraction correctly (978/12,328 = 7.93%) and notes the count matches the L1000 landmark gene set, but doesn't flag that flag_b1 and flag_b2 are overlapping indicators. The reviewer agent does flag flag_b1/flag_b2 as competing classifiers in review_findings.json, but the warning is not strong enough to make the agent surface partition-overlap in its answer.
  • mimic_se_03 (null_sentinel on glucose) — Agent reports 135.82 mg/dL across 15,144 rows, surfaces specimen mixing (CSF vs serum vs urine) and a urine-dipstick scale-mix, but never filters the VALUENUM=0 placeholder convention specific to LABEVENTS. The enrichment doesn't currently encode "zero means missing for lab values".

Both misses share a pattern: the agent computes the right number, picks up a real adjacent issue, but doesn't surface the specific trap planted in the benchmark. Both could be addressed by extending review/trap heuristics — partition-overlap detection on co-occurring boolean flags, and zero-as-sentinel detection on lab-value distributions.

Method notes

  • Same agent, same prompt template. Both arms use the standard /ask workflow: read enrichment (or skip it for blind), sample data, write Python, execute, sanity-check, emit a four-section answer. The only difference is whether .semantic-enricher/ exists in the data directory the agent reads.
  • Scoring. An LLM judge reads the agent's answer, the question, and the expected_method, and emits PASS/FAIL with a rationale. PASS requires either the right answer with the trap flagged, or an appropriate refusal grounded in the trap. A "right number with no trap caveat" is a FAIL.
  • Blind ≠ unaware. The blind agent still has access to the raw data and unrestricted reasoning. It just doesn't get the enrichment bundle. So this is not a comparison of "no knowledge vs full knowledge"; it's "training-data prior alone vs training-data prior + dataset-specific structured metadata".
  • No priming in the prompt. The /ask prompts in this run contain only the question and the data directory. No hints about the trap, the expected method, or the trap_findings file's contents.
